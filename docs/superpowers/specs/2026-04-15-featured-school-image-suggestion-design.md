# 学校图片候选抓取设计

## 背景

当前后台已经支持为 featured 学校手动维护 `hero_image_url`，并且围绕“缺图学校”建立了完整的运营视图、筛选器和排期优先级。

剩余最明显的效率缺口是：运营仍需要离开后台自行寻找学校展示图，再把图片 URL 粘贴回来。这个流程可行，但效率偏低，尤其是在首页轮换和指定日期排期已经比较完善的前提下，图片来源仍然完全依赖人工查找。

这次设计的目标不是做一个完整的图片抓取系统，而是补一个低风险的“候选抓图辅助”能力，帮助运营更快找到可用图片，再由人工确认是否保存。

## 目标

- 允许后台针对单个学校手动触发一次图片候选抓取
- 优先从学校详情已有官网链接中抓取候选图
- 抓取结果只作为建议展示，不自动覆盖当前 `hero_image_url`
- 运营可以在后台直接确认并使用候选图
- 失败时给出清晰原因，继续保留现有手动填写 URL 的流程

## 非目标

- 不做批量抓图
- 不做自动保存
- 不做专业图片
- 不做多候选图图库
- 不做图片下载到本地存储
- 不做裁剪、压缩、尺寸校验
- 不做定时抓图或后台任务队列

## 方案选择

### 方案 A：只加人工快捷链接

后台只提供学校官网、搜索页等快捷入口，运营仍手动寻找图片并粘贴 URL。

优点：
- 实现最轻
- 无抓取稳定性风险

缺点：
- 运营效率提升有限
- 仍然需要频繁切出后台

### 方案 B：单次候选抓图 + 人工确认保存（推荐）

后台对某个学校触发一次候选抓图，API 返回候选图片 URL 和来源页；运营确认后再通过现有保存接口写入 `hero_image_url`。

优点：
- 风险可控
- 与现有后台流程兼容
- 能明显缩短“缺图学校”补图链路

缺点：
- 仍然需要人工确认
- 第一版抓图覆盖率取决于学校官网结构

### 方案 C：自动抓图并直接写入

系统自动从官网抓图并写回 featured 内容配置。

优点：
- 表面效率最高

缺点：
- 容易误抓
- 一旦抓错会污染前台展示
- 当前阶段不适合引入这么高的运营风险

## 推荐方案

采用方案 B。

核心原则：
- 只为单个学校生成一个候选图
- 候选图不自动保存
- 当前图片永远优先，候选结果只是辅助
- 抓图失败不影响现有人工流程

## 架构与边界

### API 边界

新增一个 admin 端点：

- `POST /api/admin/featured-content/schools/{slug}/suggest-image`

职责：
- 校验学校是否存在
- 读取学校详情里的官网链接
- 请求官网首页 HTML
- 从页面中提取最可能用作展示图的候选 URL
- 返回候选结果，不持久化任何数据

### Web 边界

在现有 `/admin` 的学校配置区中，为每个学校配置行增加：

- `尝试抓取图片`
- 当抓取成功时显示候选图预览
- `使用该图片`

“使用该图片”继续复用现有更新 featured 学校配置的 action，不新增新的保存链路。

### 数据边界

这一步不改 `featured-content.json` 的结构。

候选抓图结果只存在于本次请求响应中，不单独入库、不写配置文件，只有运营点击“使用该图片”后，才会通过现有学校更新逻辑把 URL 写进 `hero_image_url`。

## 抓图来源策略

第一版只抓学校详情中已有的官网地址。

顺序：
1. 从 catalog 中读取该学校详情
2. 找到官网链接字段
3. 请求官网首页 HTML
4. 提取图片候选

不做：
- 搜索引擎抓图
- 第三方图床抓取
- 多页面深爬
- 多候选排序系统

这样可以保持来源可解释，也更容易让运营判断这张图是否合适。

## 候选图片提取规则

第一版规则保持简单可维护：

优先尝试以下来源：
- Open Graph 图：`meta[property="og:image"]`
- `meta[name="og:image"]`
- 页面里首个可解析的绝对或相对图片链接

对于相对路径：
- 使用官网页面 URL 解析成绝对 URL

对于提取失败：
- 返回明确失败状态，不做隐式兜底

第一版不做复杂启发式评分。只要找到一个合理候选，就返回单个候选图。

## API 返回形状

建议新增一个最小响应模型：

```json
{
  "slug": "southeast-university",
  "name": "东南大学",
  "status": "found",
  "source_url": "https://www.seu.edu.cn/",
  "suggested_image_url": "https://www.seu.edu.cn/assets/hero.jpg",
  "message": null
}
```

状态建议：
- `found`
- `missing`
- `failed`

返回语义：
- `found`：找到了候选图
- `missing`：学校没有官网，或官网里没有可用图片
- `failed`：请求或解析过程失败

## 后台交互设计

每个学校配置行新增：

- `尝试抓取图片`
- 若抓取成功，显示：
  - 候选图缩略预览
  - 来源页链接
  - `使用该图片`

行为：
- `尝试抓取图片` 提交后刷新当前 `/admin`
- 候选结果只显示在对应学校行下方
- `使用该图片` 直接调用现有更新 featured 学校 action，把候选图 URL 写入 `hero_image_url`
- 如果学校当前已经有图，也允许尝试抓取，但候选图不会自动覆盖当前图

## 失败处理

失败状态建议统一成三类：

- `学校未配置可抓取的官网地址`
- `官网页面未找到可用图片`
- `抓取失败，请稍后重试`

要求：
- 失败时不覆盖当前图片
- 失败时不写入任何配置
- 失败信息仅作用于当前学校行
- 其它后台区域不受影响

## 测试策略

### API 测试

覆盖：
- 学校存在且官网可解析图片时，返回 `found`
- 学校没有官网时，返回 `missing`
- 官网页面无图时，返回 `missing`
- 抓取异常时，返回 `failed`
- 未知学校 slug 时返回 `404`

HTTP 请求应通过 mock/stub 完成，不依赖真实外网。

### Web 测试

覆盖：
- admin 页面渲染 `尝试抓取图片`
- 抓取成功时展示候选图和 `使用该图片`
- 抓取失败时展示明确提示
- 点击 `使用该图片` 会走现有学校更新动作

不做浏览器级端到端抓图测试。

## 风险与约束

- 学校官网结构不统一，第一版命中率不会是 100%
- 某些官网图片可能不是最适合首页展示的图，因此必须保留人工确认
- 外部网站响应慢或阻断访问时，会导致抓图失败，但这不应影响现有后台手工流程

## 验收标准

- 后台学校配置行可以手动触发候选抓图
- API 能返回候选图、来源页和明确状态
- 抓图成功时，运营可一键将候选图写入现有 `hero_image_url`
- 抓图失败时，后台能显示原因且不改动当前配置
- API 测试、Web 测试和 `next build` 均通过
